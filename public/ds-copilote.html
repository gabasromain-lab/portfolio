00 / About

How we
built it.

A design system isn't a moodboard — it's a contract between design and engineering. Copilote was built bottom-up from raw values, structured into layered tokens, named by category, and stress-tested against real product surfaces before being released as a library.

We didn't start with components. We started with the smallest unit a designer can disagree about — a hex, a pixel, a px-radius — and worked up from there.

Design Platform · Kick-off note, Q3 2024

01
Tokens before components

Every visual decision lives in a token. Components consume tokens; they never hard-code values. This makes a re-skin a config change, not a redesign.

02
One scale per dimension

A single ramp for space, a single ramp for radius, a single ramp for elevation. No bespoke values. If a value isn't on the scale, it doesn't exist.

03
Names describe role, not look

Semantic tokens (bg.surface, fg.muted) sit on top of raw color steps. Designers reach for the semantic; the raw is an implementation detail.

04
8px is the law

Every spacing, sizing, and type leading value snaps to an 8px baseline. Vertical rhythm across screens stays consistent without anyone having to think about it.

05
Light first, dark in parity

Surfaces, elevations and ink were designed for light, then mirrored on the Asphalt scale for dark. Every dark value has a light twin and vice versa.

Three files, one library

The system ships as three Figma files, each owning a single layer of the stack. They are linked together as a library — components reference foundations, screens reference components, and a token edit propagates through every published file in one update.

Architecture · Three-tier tokens

Layered, never flat

L1
Primitives
Raw values. The whole alphabet of the system. Designers never reach for these directly.
asphalt-500: #1D1D26
green-200: #23D63E
space-large: 32px
L2
Semantic
Role-based aliases. What the value means in a UI, not what it looks like. This is the layer designers and engineers use.
bg.surface → white
fg.default → asphalt-500
fg.brand → green-200
L3
Component
Component-scoped tokens that resolve to semantics. Lets a component evolve its surface without leaking into the global vocabulary.
btn.primary.bg → fg.default
btn.primary.fg → fg.inverse
input.border → border.default

Why three layers?

A flat token set ages poorly. The day you decide brand green is now slightly more saturated, you don't want to chase down every component referencing green-200 — you want to update one alias.

The semantic layer is the contract. Primitives can be retuned, components can be rebuilt; as long as fg.brand still means "the green our brand uses", everything downstream keeps working.

Component tokens are the escape hatch. When a single component needs a value the semantic layer doesn't yet expose, we add a component-scoped token rather than reaching into primitives.

Convention · Naming

Predictable by design

category · family · step
Category What kind of token The dimension the token belongs to. Always one word, lowercase, plural where natural.
color space radius elevation font
Family Which group inside The named subset within the category. Brand-specific where the brand has opinions, generic where it doesn't.
asphalt green red small x1
Step Where on the scale A number for color steps (5–950, lower = lighter), or a t-shirt size for spacing & radii, or omitted when the family is itself the step.
5 / 50 / 500 tiny → massive xs → xl
Trade-offs · What we chose & why

Decisions on the record

Decided
Asphalt instead of pure black for ink

Pure black on white is harsh and rarely the right choice on screen. Asphalt-500 (#1D1D26) carries a hint of blue, sits softer against white, and pairs naturally with the brand green.

Affects · all body text, all borders, all dark surfaces
Decided
Kanit as the single typeface

One family, five weights, used from 11px legal text up to 110px display. A single typeface kept the system tight and made type pairing a non-question. Roboto Mono for tokens and code only.

Affects · 26 type styles, all components
Decided
T-shirt sizes for space, numeric for color

Spacing names (tiny, small, large) read intuitively in conversation. Color steps stay numeric (50, 100, 500) because designers do reason about lightness as a number — and the numeric ordering survives palette extension.

Affects · token names across the system
Rejected
A separate dark-mode token set

We considered color.bg.surface.dark shadow tokens. Rejected — the Asphalt scale already encodes the dark surfaces. Themes flip the semantic alias, not the token set. One source of truth, two appearances.

Trade · simplicity over theme flexibility
Rejected
Per-component spacing scales

Tempting to give Cards their own padding ramp. Rejected — once one component gets a private scale, every component asks. The 9-step global ramp is enough; constraint forces consistency.

Trade · uniformity over per-component freedom
Rejected
Free-form drop shadows

Five elevation steps. No offset, no spread, no per-component shadow values. Depth is communicated by intensity, not by a custom recipe each time. Anything that needs a different shadow needs a design conversation, not a new value.

Trade · clarity over expressive depth

04 / Foundations

8 px baseline.

All vertical metrics — line-heights, paddings, component heights — round to an 8 px multiple. Type scales align to the grid via half-leaded line boxes.

Aligned to 8 px

Each line of body text occupies a multiple of 8 px so paragraphs stack predictably alongside form controls, list rows, and tables. The visible blue grid tracks every increment.
